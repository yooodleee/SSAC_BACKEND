# ADR 003: 핵심 원칙 #2 — 환경 중심 설계

**상태**: 채택됨  
**날짜**: 2026-04-18  
**담당자**: 팀 전체

---

## 컨텍스트

AI 에이전트는 긴 텍스트 지시보다 **구조화된 환경**에서 더 안정적으로 동작한다.
단일 지시 파일(예: AGENTS.md)에 모든 규칙을 적으면 다음 문제가 발생한다:

- 에이전트가 파일 전체를 읽지 않거나 오래된 내용을 무시한다
- 규칙이 기계적으로 강제되지 않아 위반이 런타임에야 발견된다
- 파일이 길어질수록 신뢰도가 떨어지고 유지보수가 어려워진다

---

## 결정 사항

### 핵심 원칙 #2: 환경 중심 설계 (Environment-Centric Design)

**긴 프롬프트가 아닌 구조화된 환경을 기반으로 동작해야 한다.**
**단일 지시 파일에 의존하지 않는다.**

---

## 원칙의 의미

### "환경"이란 무엇인가

환경은 에이전트가 코드를 작성하는 맥락 전체를 의미한다:

| 환경 구성 요소 | 예시 | 역할 |
|--------------|------|------|
| **빌드 도구** | Gradle, `build.gradle` | 의존성, 컴파일 규칙 |
| **정적 분석** | Checkstyle (`config/checkstyle/`) | 스타일 규칙 자동 강제 |
| **아키텍처 테스트** | ArchUnit (`ArchitectureTest.java`) | 레이어 의존성 자동 강제 |
| **CI 파이프라인** | `.github/workflows/ci.yml` | 빌드/테스트/lint 자동 실행 |
| **구조화된 문서** | `docs/architecture.md`, `docs/conventions.md` | 역할별로 분리된 규칙 파일 |
| **ADR** | `docs/decisions/` | 이미 결정된 사항의 영구 기록 |

### "프롬프트 중심"이란 무엇인가 (금지 패턴)

- AGENTS.md 한 파일에 모든 규칙을 나열하는 것
- "반드시 X를 해라"는 텍스트 지시에만 의존하는 것
- 규칙 위반을 코드 리뷰(인간 눈)로만 감지하는 것
- 설정 파일 없이 구두/텍스트 합의에만 의존하는 것

---

## 설계 기준: 규칙 배치 원칙

모든 규칙은 아래 우선순위에 따라 배치한다:

```
1순위: 도구로 강제  →  ArchUnit, Checkstyle, CI 빌드 실패
2순위: 구조로 강제  →  패키지 분리, 파일 분리, 설정 파일
3순위: 문서로 안내  →  docs/architecture.md, docs/conventions.md
4순위 (지양): 지시로 요청  →  AGENTS.md 텍스트 나열
```

> AGENTS.md는 4순위 수단이다. 1~3순위로 표현할 수 없는 것만 AGENTS.md에 기록한다.

---

## AGENTS.md의 올바른 역할

AGENTS.md는 **단일 지시 파일이 아닌 진입점 인덱스**다.

올바른 AGENTS.md:
- 어떤 파일을 읽어야 하는지 포인터를 제공한다
- 핵심 원칙을 한 줄 요약 + ADR 링크로 안내한다
- 커밋 전 체크리스트처럼 자동화가 불가능한 절차를 기록한다

잘못된 AGENTS.md:
- 아키텍처 규칙을 직접 상세히 나열한다 (→ `docs/architecture.md`에 위임)
- 네이밍 컨벤션을 모두 작성한다 (→ `docs/conventions.md`에 위임)
- Checkstyle로 강제 가능한 규칙을 텍스트로 반복한다

---

## 문서 분리 원칙

하나의 `docs/` 파일은 하나의 관심사만 다룬다:

| 파일 | 단일 관심사 |
|------|------------|
| `docs/architecture.md` | 레이어 구조와 의존성 방향만 |
| `docs/conventions.md` | 네이밍과 코드 스타일만 |
| `docs/quality.md` | 기술 부채와 품질 현황만 |
| `docs/onboarding.md` | 환경 설정과 초기 세팅만 |
| `docs/decisions/` | 결정 사항 영구 기록만 |

새로운 규칙을 추가할 때 AGENTS.md에 직접 쓰기 전에 먼저 물어라:
> "이 규칙을 도구(ArchUnit/Checkstyle/CI)로 강제할 수 있는가?"  
> "이 규칙은 기존 `docs/` 파일 중 어디에 속하는가?"

---

## 채택 이유

- 에이전트는 긴 텍스트보다 **실패하는 빌드**에 더 즉각적으로 반응한다
- 도구로 강제된 규칙은 에이전트 교체/버전 변경 시에도 유지된다
- 파일이 짧고 집중될수록 에이전트가 관련 맥락만 읽어 오염 없이 동작한다

---

## 재검토 조건

- 도구로 표현할 수 없는 규칙 유형이 반복적으로 발생할 때
- 문서 분산으로 인해 에이전트 혼란이 증가할 때 (통합 재검토)
