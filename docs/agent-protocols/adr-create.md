# ADR 생성 프로토콜 (BE)

## 역할
동일한 오류가 3회 이상 반복되거나
중요한 기술 의사결정이 발생했을 때
ADR(Architecture Decision Record)을 생성하여
의사결정 맥락과 근거를 문서화한다.

## 트리거 조건
- 동일한 오류가 3회 이상 반복 발생 시
- 기존 구현 방식을 전면 변경하는 결정 시
    예) @Cacheable → StringRedisTemplate 수동 캐싱 전환
    예) S3 → Cloudinary 이미지 저장소 전환
- 기술 스택 변경 / 라이브러리 교체 시
- 보안 정책 변경 시
- CLAUDE.md 오류 발생 시 3순위 참조 시

## 실행 순서
STEP 1. 오류 / 의사결정 맥락 파악
STEP 2. 원인 분석 및 대안 탐색
STEP 3. ADR 문서 생성
STEP 4. CLAUDE.md / 관련 프로토콜 갱신 필요 여부 판단
STEP 5. debug-log.md 기록

---

## STEP 1. 오류 / 의사결정 맥락 파악

ADR 생성 전 아래 항목을 확인한다:

### 반복 오류 트리거인 경우
□ 오류 메시지 / 스택 트레이스 수집
□ 오류 발생 횟수 및 발생 패턴 확인
    예) 동일 오류가 3회 이상 / 동일 원인 반복
□ 이전 log-diagnose.md 실행 결과 참조
□ debug-log.md에서 동일 이력 검색

### 기술 의사결정 트리거인 경우
□ 변경 전 기술 스택 / 방식 확인
□ 변경 배경 (문제점 / 한계) 확인
□ 변경 후 기술 스택 / 방식 확인
□ 변경으로 인한 영향 범위 확인

파악 완료 후 아래 형식으로 출력한다:

```
[ADR 생성 맥락]
- 트리거 유형 : 반복 오류 / 기술 의사결정
- 발생 횟수   : N회 (반복 오류인 경우)
- 핵심 문제   : {한 줄 요약}
- 영향 범위   : {관련 파일 / 서비스 / API}
```

## STEP 2. 원인 분석 및 대안 탐색

### 근본 원인 분석 (5-Why)
발생한 오류 / 문제의 근본 원인을
최소 3단계 이상 추적한다:

```
Why 1. 표면적 오류는 무엇인가?
Why 2. 왜 이 오류가 발생했는가?
Why 3. 왜 이 설계가 채택되었는가?
(Why 4. 왜 사전에 감지되지 않았는가?)
(Why 5. 왜 프로토콜에서 걸러지지 않았는가?)
```

예시:
```
Why 1. Redis 역직렬화 오류 발생
Why 2. @Cacheable이 GenericJackson2JsonRedisSerializer로
             직렬화하여 final 타입에 타입 정보가 미포함됨
Why 3. 기본 RedisTemplate 설정을 그대로 사용
Why 4. self-diagnose.md에 직렬화 점검 항목 없음
Why 5. 이전 스프린트 트러블슈팅이 프로토콜에 미반영됨
```

### 대안 탐색
채택 가능한 해결 방안을 아래 형식으로
2개 이상 나열한다:

| 대안 | 장점 | 단점 | 결론 |
|-----|-----|-----|-----|
| A   | ... | ... | 채택 / 미채택 |
| B   | ... | ... | 채택 / 미채택 |

## STEP 3. ADR 문서 생성

아래 경로에 ADR 파일을 생성한다:
```
docs/adr/ADR-{번호}-{제목-kebab-case}.md
```

번호 규칙:
```
→ 기존 ADR 파일 중 가장 큰 번호 + 1
→ 첫 번째 ADR: ADR-001
→ 예) ADR-003-redis-serialization-strategy.md
```

### ADR 문서 양식
```markdown
---
# ADR-{번호}: {제목}

## 맥락 (Context)
이 결정이 필요했던 배경과 문제 상황을 서술한다.
→ 어떤 문제가 발생했는가?
→ 왜 지금 결정이 필요한가?

## 결정 (Decision)
채택한 해결 방안과 그 이유를 서술한다.
→ 무엇을 선택했는가?
→ 왜 이 방안을 선택했는가?

## 결과 (Consequences)
이 결정이 가져오는 영향을 서술한다.
→ 긍정적 영향
→ 부정적 영향 / 트레이드오프
→ 향후 검토 필요 항목

## 프로토콜 반영 필요 여부
□ self-diagnose.md 갱신 필요 여부
□ sc-structure-check.md 갱신 필요 여부
□ testing.md 갱신 필요 여부
□ CLAUDE.md 갱신 필요 여부
□ flyway.md 갱신 필요 여부

## 작성일
{YYYY-MM-DD}

## 작성자
{에이전트 / 팀원}
---
```

## STEP 4. 프로토콜 갱신 필요 여부 판단

ADR 작성 완료 후 아래 기준으로
관련 프로토콜 갱신 필요 여부를 판단한다:

### 판단 기준

**self-diagnose.md 갱신 필요 조건:**
→ 자가 진단에서 걸러지지 않은 오류인 경우
→ 새로운 점검 항목이 필요한 경우
예) 캐싱 직렬화 오류 → 직렬화 점검 항목 추가

**sc-structure-check.md 갱신 필요 조건:**
→ 구조 충돌 점검에서 감지하지 못한 패턴인 경우
예) MySQL 미지원 문법 → 점검 항목 추가

**testing.md 갱신 필요 조건:**
→ 테스트로 사전 감지 가능했던 오류인 경우
예) 커버리지 기준 미흡으로 0% 클래스 은폐

**CLAUDE.md 갱신 필요 조건:**
→ 전역 금지 규칙으로 등록해야 할 패턴인 경우
예) ADD COLUMN IF NOT EXISTS MySQL 미지원 금지

**flyway.md 갱신 필요 조건:**
→ 마이그레이션 컨벤션 위반 패턴인 경우

### 갱신 출력 형식
```
[프로토콜 갱신 필요 항목]
✅ self-diagnose.md → 캐싱 직렬화 점검 항목 추가 필요
✅ CLAUDE.md        → 갱신 불필요
✅ testing.md       → 클래스별 커버리지 임계값 추가 필요
```

→ 갱신이 필요한 항목은 별도 백로그 SC로 등록한다.
→ 이 프로토콜에서 직접 수정하지 않는다.

## STEP 5. debug-log.md 기록

ADR 생성 완료 후 docs/debug-log.md에
아래 형식으로 기록한다:

```markdown
## [YYYY-MM-DD] ADR 생성

### ADR 정보
- 파일명: ADR-{번호}-{제목}.md
- 트리거: 반복 오류 N회 / 기술 의사결정
- 핵심 결정: {한 줄 요약}

### 프로토콜 갱신 필요 항목
- {프로토콜 파일명} → {갱신 내용}

### 백로그 SC 등록 필요 항목
- {항목}

### 완료 시각
{datetime}
```
