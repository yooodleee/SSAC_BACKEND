# ADR 생성 프로토콜 (BE)

## 역할
동일 오류 3회 이상 반복 또는 중요 기술 의사결정 시 ADR을 생성하여 맥락과 근거를 문서화한다.

## 트리거 조건
- 동일 오류 3회 이상 반복 발생
- 기존 구현 방식 전면 변경 (예: `@Cacheable` → `StringRedisTemplate` 수동 캐싱, S3 → Cloudinary)
- 기술 스택 변경 / 라이브러리 교체 / 보안 정책 변경

## 실행 순서
STEP 1(맥락 파악) → 2(원인 분석 및 대안 탐색) → 3(ADR 문서 생성) → 4(프로토콜 갱신 판단) → 5(기록)

---

## STEP 1. 오류 / 의사결정 맥락 파악

**반복 오류인 경우**
- [ ] 오류 메시지 / 스택 트레이스 수집
- [ ] 오류 발생 횟수 및 패턴 확인
- [ ] `debug-log.md`에서 동일 이력 검색

**기술 의사결정인 경우**
- [ ] 변경 전 기술 스택 / 방식 및 문제점 확인
- [ ] 변경 후 기술 스택 / 방식 및 영향 범위 확인

---

## STEP 2. 원인 분석 및 대안 탐색

**5-Why 근본 원인 분석** (최소 3단계):
```
Why 1. 표면적 오류는 무엇인가?
Why 2. 왜 이 오류가 발생했는가?
Why 3. 왜 이 설계가 채택되었는가?
Why 4. 왜 사전에 감지되지 않았는가?
Why 5. 왜 프로토콜에서 걸러지지 않았는가?
```

**대안 탐색** (2개 이상):

| 대안 | 장점 | 단점 | 결론 |
|-----|-----|-----|-----|
| A   | ... | ... | 채택 / 미채택 |
| B   | ... | ... | 채택 / 미채택 |

---

## STEP 3. ADR 문서 생성

파일 위치: `docs/adr/ADR-{번호}-{제목-kebab-case}.md`  
번호: 기존 ADR 중 최대 번호 + 1 (`Glob: docs/adr/ADR-*.md`로 확인)

```markdown
---
# ADR-{번호}: {제목}

## 맥락 (Context)
이 결정이 필요했던 배경과 문제 상황.

## 결정 (Decision)
채택한 해결 방안과 그 이유.

## 결과 (Consequences)
- 긍정적 영향:
- 부정적 영향 / 트레이드오프:
- 향후 검토 필요 항목:

## 프로토콜 반영 필요 여부
- [ ] self-diagnose.md
- [ ] sc-structure-check.md
- [ ] testing.md
- [ ] CLAUDE.md
- [ ] flyway.md

## 작성일
{YYYY-MM-DD}
---
```

---

## STEP 4. 프로토콜 갱신 필요 여부 판단

| 프로토콜 | 갱신 필요 조건 |
|---------|------------|
| `self-diagnose.md` | 자가 진단에서 걸러지지 않은 오류 / 신규 점검 항목 필요 |
| `sc-structure-check.md` | 구조 충돌 점검에서 감지 못한 패턴 |
| `testing.md` | 테스트로 사전 감지 가능했던 오류 |
| `CLAUDE.md` | 전역 금지 규칙으로 등록할 패턴 |
| `flyway.md` | 마이그레이션 컨벤션 위반 패턴 |

> 갱신이 필요한 항목은 백로그 SC로 등록한다. 이 프로토콜에서 직접 수정하지 않는다.

---

## STEP 5. debug-log.md 기록

```
## ✅ [ADR] YYYY-MM-DD — ADR-{번호}: {제목}

### ADR 정보
- 트리거: 반복 오류 N회 / 기술 의사결정
- 핵심 결정: {한 줄 요약}

### 프로토콜 갱신 필요 항목
- {프로토콜 파일명} → {갱신 내용} / 없음

### 백로그 SC 등록 필요 항목
- {항목} / 없음
```
