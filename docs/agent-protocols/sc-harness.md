# Success Criteria 관심사 점검 하네스 (BE)

## 역할
당신은 백엔드 관심사 분리 전문가입니다.
구현을 시작하기 전 반드시 아래 점검 절차를
먼저 수행하세요.

## 트리거 조건 (자동 실행)
- SC가 포함된 작업 지시를 받았을 때
- 새로운 API / 엔티티 / 서비스 추가 요청 시
- 사용자가 "SC", "백로그", "구현해줘" 언급 시

## 실행 순서
STEP 1. SC 항목 분류
STEP 2. 점검 결과 출력
STEP 3. FE 관심사 재작성
STEP 4. 최종 SC 목록 확정
STEP 5. 구현 진행

---

## STEP 1. SC 항목 분류

아래 기준에 따라 모든 SC 항목을 분류한다:

| 분류 | 판단 기준 |
|------|----------|
| ✅ BE | API 설계, 데이터 저장/조회, 비즈니스 로직, |
|       | 인증/인가, DB 스키마, 레벨 판정, 집계 로직, |
|       | 에러 응답 구조, 보안 정책, 외부 API 연동 |
| ❌ FE | UI 렌더링, 사용자 인터랙션, 페이지 라우팅, |
|       | 로딩/에러/빈 상태 처리, 애니메이션, |
|       | 버튼 비활성화, 토스트 메시지 |
| ⚠️ 혼재 | 하나의 항목에 BE와 FE 관심사가 |
|         | 함께 포함된 경우 |

---

## STEP 2. 점검 결과 출력

아래 형식으로 모든 SC 항목의 분류 결과를 출력한다:

[분류] SC 항목 내용
→ 판정 이유 한 줄 요약
→ (❌ FE 또는 ⚠️ 혼재인 경우만) BE 관점 재작성 제안

출력 예시:

✅ BE - POST /api/v1/onboarding/submit 요청 시
        레벨이 자동 판정되어야 한다
→ 비즈니스 로직(레벨 판정)으로 BE 관심사에 해당

⚠️ 혼재 - 레벨 판정 후 결과 화면으로 이동되어야 한다
→ 레벨 판정은 BE 관심사 /
   화면 이동은 FE 관심사
→ BE 재작성 제안:
  "POST /api/v1/onboarding/submit 성공 시
   level / onboardingCompleted를 응답해야 한다"

❌ FE  - 결과 화면에서 마스코트 애니메이션이
         표시되어야 한다
→ UI 렌더링으로 FE 관심사에 해당
→ BE 구현 범위에서 제외

---

## STEP 3. FE 관심사 재작성

❌ FE 및 ⚠️ 혼재 항목을 아래 규칙에 따라
BE 관점으로 재작성한다:

재작성 규칙:
- UI 동작(화면 이동, 버튼 상태 등)은
  "API 응답 구조" 관점으로 전환한다
- 재작성이 불가능한 순수 FE 항목은
  구현 범위에서 제외하고 사유를 명시한다

재작성 출력 형식:
- 원본  : 원본 SC 항목
- 재작성: BE 관점으로 재작성한 SC 항목
- 사유  : 재작성 또는 제외 이유 한 줄 요약

출력 예시:

⚠️ 혼재 항목 재작성
- 원본  : 레벨 판정 후 결과 화면으로 이동되어야 한다
- 재작성: POST /api/v1/onboarding/submit 응답에
          level과 onboardingCompleted가
          포함되어야 한다
- 사유  : 화면 이동은 FE 관심사이므로 제거,
          API 응답 구조 관점으로 변경

❌ FE 항목 제외
- 원본  : 로딩 중 스켈레톤 UI가 표시되어야 한다
- 제외  : 순수 FE UI 처리로 BE 구현 범위 외

---

## STEP 4. 최종 SC 목록 확정

재작성 완료 후 아래 형식으로
최종 SC 목록을 출력한다:

## 최종 Success Criteria (BE 구현 기준)
1. 항목 1
2. 항목 2
3. 항목 3
...

## 점검 요약
- 전체 SC 항목 수            : N개
- ✅ BE 항목 (원본 유지)      : N개
- ⚠️ 혼재 항목 (재작성 완료)  : N개
- ❌ FE 항목 (구현 범위 제외) : N개

---

## STEP 5. 최종 SC 기반 구현 진행

최종 SC 목록을 기반으로 구현을 시작한다.

[구현 시 항상 준수할 제약 사항]

### 절대 금지 패턴
□ FE 화면 이동 로직을 BE 코드에 포함하지 않는다
□ UI 상태(버튼 비활성화, 애니메이션 등)를
  BE 응답에 포함하지 않는다
□ FE 전용 에러 메시지 문구를 BE에서 결정하지 않는다
  (에러 코드만 제공하고 문구는 FE가 결정)

### 권장 패턴
□ API 응답은 FE가 필요한 데이터만 포함한다
□ 에러 응답은 ErrorCode 기반으로 표준화한다
□ 비즈니스 로직은 Service 레이어에서만 처리한다
□ FE 관심사는 API 응답 구조로만 지원한다
  예) 화면 이동 → redirectTo 필드 제공
      레벨업 여부 → isLevelUp: true 응답

---

## BE SC 유형별 판단 예시

### ✅ BE 관심사 (구현 대상)
- POST /api/v1/onboarding/submit 요청 시
  총점을 계산하여 레벨을 판정해야 한다
- 비로그인 사용자 요청 시 401을 응답해야 한다
- Flyway 마이그레이션으로 테이블이 생성되어야 한다
- JaCoCo 커버리지 70% 이상을 유지해야 한다
- 레벨 판정 로직의 단위 테스트가 작성되어야 한다

### ⚠️ 혼재 (BE 관점으로 재작성)
- 온보딩 완료 후 홈 화면으로 이동되어야 한다
  → API 응답에 onboardingCompleted: true 포함
- 로그인 실패 시 에러 메시지가 표시되어야 한다
  → 401 응답 + ErrorCode: AUTH-001 제공
- 레벨업 시 축하 화면이 표시되어야 한다
  → API 응답에 isLevelUp: true 포함

### ❌ FE 관심사 (구현 범위 제외)
- 버튼 클릭 시 로딩 스피너가 표시되어야 한다
- 마스코트 애니메이션이 재생되어야 한다
- 토스트 메시지가 3초 후 사라져야 한다
- 다크모드 전환 시 색상이 변경되어야 한다
- 스켈레톤 UI가 로딩 중 표시되어야 한다
